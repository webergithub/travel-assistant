## 0. 本文定位

| | 需求规格说明书① | 本文② |
|---|---|---|
| 回答 | 现在是什么（as-built） | 应该是什么、缺什么 |

核心方法论断言：**先写失效清单与验收标准，再写功能描述。**「AI 生成行程」没有信息量；「AI 推荐的地点要么被地图证实、要么被显式标记为未证实，绝不无标注混入」才是需求。本文验收标准一律破坏性写法。

## 2. 用户与任务分析（D1）

### 2.1 核心任务与成功条件

**T1 行前规划**：「把碎片攻略变成一份可信、顺路、节奏合理的逐日计划」

| 成功条件 | 反推需求 | 现状 |
|---|---|---|
| 推荐的地点真实存在 | 地图核验+显式标注 | ✅ FR-5.3 |
| 同天地点顺路 | 地图连线可视化检查 | ✅ FR-4.4（人工检查）⚠ 无自动路线优化 |
| 全程天数完整覆盖 | 生成范围=行程天数 | ⚠ KI-11：>10 天静默截断 |
| 天气影响可预判 | 逐日天气 | ✅ FR-7 |

**T2 行中执行**：「当天在手机上照着计划走，随手调整」

| 成功条件 | 反推需求 | 现状 |
|---|---|---|
| 手机上能完整编排 | 触屏跨天移动 | ❌ KI-1（品类级硬伤） |
| 无网/弱网能看行程 | 离线只读缓存 | ❌ 未实现 |
| 今天的安排一眼可见 | 当日聚焦视图 | ⚠ 需滚动长列表 |
| 一键导航到下一站 | 地图 App 深链 | ❌ 未实现 |

**T3 同行协作**：「几个人一起把行程定下来」——只读分享 ✅（FR-6），共同编辑 ❌。

**T4 行程交付**：「拿出一份能递签、能打印、能发群的行程单」——✅ FR-8（v0.2 已闭环）。

### 2.2 优先级纠偏结论

按 JTBD 检验，**T2（行中）是使用频次最高的任务却是现状最弱面**：触屏编排缺失使手机用户被降级为只读访客。「移动可用」不是体验优化，是任务前提，判 P0。同理，KI-10 类生产静默故障说明「部署后自动冒烟」是所有任务的共同前提，判 P0。

## 3. 品类规范对标（D2）

对标 Wanderlog、TripIt、穷游行程助手、Mindtrip：

| 行规能力 | 行业默认 | 本产品 | 决策建议 |
|---|---|---|---|
| 逐日编排+地图联动 | 标配 | ✅ FR-3/4 | 保持 |
| 移动端完整编辑 | 标配 | ❌ KI-1 | **P0 补齐** |
| AI 生成行程 | 常见 | ✅ 且核验防幻觉（超行规） | 保持为差异化卖点 |
| 行程导出（PDF/表格） | 常见 | ✅ FR-8 | 保持 |
| 预算：分类汇总/多币种 | 常见 | ⚠ 仅单项费用+合计 | P2 增强 |
| 行前清单（打包/待办） | 常见 | ❌ | P2 |
| 多人共同编辑 | 常见 | ❌ 仅只读分享 | P2（先邀请制协作，实时性后置） |
| 预订邮件导入（TripIt 核心） | 高端 | ❌ | 显式不做：依赖邮箱授权生态，成本收益不成比 |
| 离线访问 | 常见 | ❌ | P2（PWA 只读缓存） |
| 地点详情（开放时间/门票） | 常见 | ⚠ 仅名称/地址/坐标 | P3：受免费数据源约束，先靠 AI note 提示 |
| 实时航班状态 | 高端 | ❌ | 显式不做（v1.0 前不碰付费数据源） |

对标结论：叫得上「行程规划工具」品类名的门槛缺口只有一条——移动端编排（其余标配已达标）；常见级缺口按预算、清单、离线、协作次序补。

## 4. 场景与环境分析（D4）

设计基准 = **中国大陆网络 + 普通安卓手机浏览器**（而非开发机 Chrome）：

| 环境事实 | 影响 | 派生需求 |
|---|---|---|
| Carto/OSM 瓦片在大陆时延高或不可达 | 地图白屏=半个产品失效，且当前无任何提示 | PR-P0-5 瓦片健康检测+可切换底图（天地图/高德留接口，同 C-1 零成本原则冲突时优先国内可达性） |
| Nominatim 中文 POI 覆盖弱（餐厅/小众点常查无） | 搜索空结果、AI 核验误伤真实地点 | PR-P1-9 核验降级语义（KI-7）+搜索空结果引导换词 |
| 手机弱网 | 行程打不开 | PR-P2-4 离线只读 |
| 微信内置浏览器是国内分享链接主要打开场景 | 未实测，打印/下载行为存疑 | PR-P1-10 微信内实测并给降级提示 |

## 5. 失效模式与可观测性（D3）

核心一问：哪些失效用户当下察觉不到？FMEA-lite（全部经代码核实，证据见 Gap 文档附录 B）：

| 失效 | 用户当下可察觉? | 后果 | 派生需求 |
|---|---|---|---|
| 生产库未初始化（已发生，KI-10） | 否——health 仍绿 | 全部写操作崩进程 | PR-P0-2 部署冒烟：触库全链路 |
| 天气拉取失败 | 否——静默无天气 | 误以为无预报 | PR-P1-4 降级提示 |
| 核验遇网络故障标 FAIL（KI-7） | 否——显示「未在地图找到」 | 真实地点被冤枉，用户误删 | PR-P1-9 区分 UNVERIFIED/NOT_FOUND |
| 乐观删除后请求失败（KI-8） | 否——UI 已删 | 刷新后「诈尸」 | PR-P1-6 失败回滚 |
| >10 天 AI 截断（KI-11） | 否——草稿看起来完整 | 后段行程空白 | PR-P1-11 明示范围或分段生成 |
| 弹窗拦截打印失效（KI-6） | 是但无解释 | 以为按钮坏了 | PR-P1-4 并入降级提示 |
| 游客 token 丢失 | 否——直到想找回 | 行程永久失联 | PR-P0-4 游客升级正式账号 |
| Nominatim 全局队列被并发用户拖长 | 否——只觉得慢 | 搜索/AI 变慢无解释 | PR-P2-7 队列深度暴露+前端等待提示 |

派生的「健康自检」类需求：/api/health 扩展为触库 ping + 外部依赖探活（PR-P1-12），部署冒烟脚本读取它做门禁。

## 6. 生命周期分析（D5）

| 阶段 | 现状 | 问题 | 派生需求 |
|---|---|---|---|
| 首次接触 | 游客一键体验 ✅ | 无平台 key 时 AI 按钮 402 才发现不可用 | PR-P1-1 mock 样例行程（无 key 也能完整体验产品叙事） |
| 游客→正式 | ❌ 无迁移路径 | 换浏览器/清缓存=数据失联 | PR-P0-4 |
| 日常 | ✅ | 数据无备份，单文件库 | PR-P0-3 每日备份+异地拉取 |
| 更新 | 手动 rsync+restart | schema 变更漏 db push（已踩） | PR-P0-2 冒烟含 db 探针；发布 SOP 写入 §8.2 |
| 退役/删除 | ❌ 无删号与数据导出 | 合规与信任缺口 | PR-P2-8 数据导出（JSON）+删除账号 |

## 7. 威胁与滥用建模（D6）

| 资产 | 威胁 | 现状缓解 | 缺口 |
|---|---|---|---|

## 9. P0 缺口专项

**品类级硬伤点名：PR-P0-1（触屏编排）**——手机是行程工具的主要使用设备，编排不可用则「行程规划工具」名不副实。

| 项 | PR-P0-1 触屏编排 |
|---|---|
| 任务影响 | T2 全瘫：手机用户无法把清单项排进天 |
| 方案概要 | 卡片增加「移动到…」弹出菜单（全端可用，先行）；二期再加长按拖拽 |
| 破坏性验收 | 在 iOS Safari 与安卓 Chrome 触屏设备上，不借助键鼠把想去清单项移到第 3 天、再移回清单，全程无失败；期间桌面拖拽行为不回归 |
| 量级 | 小（菜单方案 1 轮） |

| 项 | PR-P0-2 部署冒烟 |
|---|---|
| 任务影响 | 防 KI-10 复发：静默生产故障 |
| 方案概要 | deploy.sh 收尾执行 smoke.sh：guest→POST trip→POST item→share→GET share→比对内容；任一步非 2xx 即非零退出并打印 pm2 日志尾 |
| 破坏性验收 | 人为把服务器 prod.db 改名后跑部署，冒烟必须失败且明确指出触库步骤；恢复后必须通过 |
| 量级 | 小 |

| 项 | PR-P0-3 备份 |
|---|---|
| 破坏性验收 | 删除服务器 prod.db，用最近一次备份恢复后，恢复前创建的行程完整可读；备份文件在本地也存有副本 |
| 量级 | 小（cron + sqlite .backup + scp 拉取） |

| 项 | PR-P0-4 游客升级 |
|---|---|
| 破坏性验收 | 游客建 2 个行程→升级填邮箱密码→退出→换浏览器登录该邮箱，2 个行程完整在列；原游客 token 同时失效或指向同一账号 |
| 量级 | 中（账号合并语义） |

| 项 | PR-P0-5 地图可用性 |
|---|---|
| 破坏性验收 | 用 DevTools 封锁 basemaps.cartocdn.com 后加载编辑器，10 秒内界面出现「地图源不可达已切换/请重试」提示且备用源可渲染；不封锁时无任何多余请求 |
| 量级 | 中 |

## 11. 验收与质量策略（D7）

- 可测性夹具需求：内存 SQLite 的 API 测试环境；mock LLM Provider（固定 JSON 样例，兼作 PR-P1-1 演示数据）；Nominatim/Open-Meteo 录制回放夹具（避免测试打真实接口违反限速）。
- 破坏性验收写法示范（本文所有 P0 已按此写）：描述如何制造失效，断言正确行为，禁止「功能正常」式验收。
- 现有质量手段（tsc 门禁+人工走查）定位为兜底，非替代：KI-10 证明人工冒烟覆盖不了触库路径。

## 12. 非目标与已知取舍

| 不做 | 原因 |
|---|---|
| 预订邮件导入 | 依赖邮箱授权与解析生态，个人项目成本收益不成比 |
| 实时航班/火车状态 | 免费数据源无可靠供给 |
| 机酒比价与预订分佣 | 商业形态不同，引入合规与利益冲突 |
| 实时协同（CRDT/OT） | 先做邀请制非实时协作验证需求，再评估 |
| 原生 App | Web 优先；PWA 覆盖离线诉求 |
