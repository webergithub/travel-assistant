## 2. 产品概述

### 2.1 产品定位

面向中文用户的行程规划 Web 工作台，差异化定位是「AI 生成 + 地图核验」：AI 只做草稿输入方式，结构化行程数据才是主体；每个 AI 推荐地点必须经地图核验并显式标注真伪状态，直面 AI 旅行工具最大痛点（幻觉地点）。

核心场景：新建行程 → AI 生成草稿或手动搜索添加地点 → 拖拽逐日编排（地图联动）→ 查看天气 → 分享只读链接 / 导出打印行程单。

### 2.2 目标用户与典型场景

| 用户 | 场景 |
|---|---|
| 自由行规划者 | 行前把攻略碎片整理成逐日计划，导出递签/打印行程单 |
| 结伴出行发起人 | 规划后发只读链接给同行人传阅 |
| 尝鲜访客 | 游客一键体验，无需注册 |

### 2.3 设计约束与原则

| 编号 | 约束 | 代价与理由 |
|---|---|---|
| C-1 | 外部服务零成本优先：Nominatim 地理编码、Open-Meteo 天气、Carto 底图均免 key | 换来 Nominatim 1 req/s 限速与中文 POI 覆盖弱；地图源可替换性预留 |
| C-2 | AI 输出必须经地图核验后才呈现给用户 | 生成耗时增加约 1 秒/地点，换取防幻觉的产品可信度 |
| C-3 | 沿用 OPC Studio 模块形态：Express 同进程托管前端、nginx 子路径反代、HashRouter、共享语言 key `opcstudio_lang` | 与主站部署/体验一致 |
| C-4 | 游客零门槛：一键匿名账号，数据存云端 | 游客 token 丢失即数据失联（见 KI 与规划 PR-P0-4） |
| C-5 | SQLite 单文件库 | 规模预期为个人/小群体；生产切 Postgres 留有 datasource 接口 |
| C-6 | LLM 凭证用户自带优先（`x-user-api-key`），平台 `ANTHROPIC_API_KEY` 为可选兜底 | 平台不配 key 时 AI 功能返回 402，其余功能不受影响 |

## 3. 系统架构

### 3.1 角色

| 角色 | 职责 |
|---|---|
| React SPA（frontend/） | 页面、编排交互、地图渲染、导出生成 |
| Express API（backend/src/） | 鉴权、行程 CRUD、外部服务代理（地理编码/天气/LLM）、静态托管前端 |
| SQLite（Prisma） | User / Trip / ItineraryItem / PlaceCache 四表 |
| Nominatim（OSM） | POI 搜索与 AI 地点核验（服务端代理 + 缓存 + 串行限速） |
| Open-Meteo | 逐日天气预报（服务端代理 + 缓存） |
| Anthropic API | 行程草稿生成（默认模型 claude-opus-4-8，环境变量可换） |
| Carto CDN | 暗色地图瓦片（前端直连） |

### 3.2 拓扑与数据流

浏览器 → nginx（opcstudio.cc/travel/ 反代 127.0.0.1:3500）→ Express。Express 静态托管 `public/`（前端构建产物），`/api/*` 之外的路径回退 index.html 配合 HashRouter。外部服务全部由服务端代理转发（地图瓦片除外，前端直连 Carto）。

### 3.3 安全模型

- 鉴权：JWT（HS256，90 天过期）放 `Authorization: Bearer`；密码 bcrypt 10 轮。
- 全局 `optionalAuth` 解析身份，行程/AI 路由 `requireAuth` 强制登录；行程属主校验失败一律返回 404 不泄露存在性。
- BYO Key 走请求头透传至 Anthropic，不写库不打日志；支持 `sk-ant-api*` 与 `sk-ant-oat*`（OAuth 订阅令牌自动加 beta 头）。
- 分享链接 slug 为 12 位十六进制随机串（48 bit 熵），只读接口不暴露属主邮箱。
- 前端输出经 React 转义；打印视图自实现 `esc()` 转义后再 `document.write`。

| 编号 | 需求 |
|---|---|

### 4.3 行程项与逐日编排（FR-3）

| 编号 | 需求 |
|---|---|

| FR-3.5 | 同天内 ↑↓ 按钮交换相邻项次序；排序变更批量提交（单次 ≤300 条 move） |

### 4.4 地点搜索与地图（FR-4）

| 编号 | 需求 |
|---|---|
| FR-4.1 | 搜索框（回车或 🔍 按钮提交）经服务端代理查 Nominatim，返回至多 6 条；关键词未含目的地时自动追加目的地词 |
| FR-4.2 | 服务端串行队列限速 ≥1.1s/请求，结果落 PlaceCache 缓存 30 天（读取侧判过期） |
| FR-4.3 | 结果点击定位地图，「＋想去」一键把地点（含坐标、地址、核验=OK）加入清单 |
| FR-4.4 | 地图：Carto 暗色底图；有坐标的项按天着色编号标记（想去清单灰色★），同天项连线；地点集合变化自动 fitBounds（编辑文字不触发） |
| FR-4.5 | 点击卡片定位地图，点击标记弹出名称+地址；坐标缺失的项不上图 |

### 4.5 AI 行程草稿与防幻觉核验（FR-5）

| 编号 | 需求 |
|---|---|
| FR-5.1 | 输入目的地/天数/偏好（≤500 字）/语言，LLM 输出严格 JSON（逐日 theme+items，含 searchQuery 检索词与实用 tips ≤8 条） |
| FR-5.2 | Prompt 硬约束：只推荐真实知名地点、地理顺路、每天 3–5 项含本地餐饮、cost 为人民币估算 |
| FR-5.3 | 生成后服务端逐项核验：SIGHT/FOOD/HOTEL 类型拿 searchQuery 查 Nominatim，命中→verified=OK 并回填坐标地址，未命中→FAIL；单次核验上限 40 项，超出部分标 NONE |
| FR-5.4 | 草稿不落库，抽屉内预览（含 ✓/⚠ 标注与 tips）后由用户选择「追加」或「替换」（替换清空已排期项、保留想去清单，需确认）批量写入 |
| FR-5.5 | 凭证解析次序：请求头 `x-user-api-key` 优先于平台环境变量；均无→402 `NO_API_KEY`，前端提示配置 🔑；AI 返回非法 JSON→502 `BAD_AI_JSON` 提示重试 |
| FR-5.6 | AI 请求天数上限 10 天（zod）；前端对更长行程传 min(days,10)——超出部分静默不生成，见 KI-11 |

## 6. 接口与协议规格

### 6.1 REST 端点（前缀 /api）

| 端点 | 方法 | 鉴权 | 说明 |
|---|---|---|---|
| /health | GET | 无 | 返回 ok 与平台 key 配置状态 |
| /auth/register /auth/login /auth/guest | POST | 无 | 返回 token+user；409 邮箱已注册 / 401 密码错误 |
| /auth/me | GET | 必须 | 回显身份 |
| /trips | GET, POST | 必须 | 列表（含 items 计数）/ 创建 |
| /trips/:id | GET, PUT, DELETE | 必须+属主 | 详情（含全部 items）/ 更新 / 删除；非属主 404 |
| /trips/:id/share | POST | 必须+属主 | body.enable 开关分享，返回 shareSlug |
| /trips/:id/items | POST | 必须+属主 | 新增项，sortOrder 取该天末尾 |
| /trips/:id/items/bulk | POST | 必须+属主 | mode 取 append / replace，items ≤200，事务写入 |
| /trips/:id/items/:itemId | PUT, DELETE | 必须+属主 | 更新 / 删除单项 |
| /trips/:id/reorder | POST | 必须+属主 | moves ≤300 条，事务批量改 dayIndex/sortOrder |
| /places/search | GET | 无 | q ≤120 字，返回 ≤6 条 {name, displayName, lat, lng, kind} |
| /share/:slug | GET | 无 | 只读行程+items+分享者昵称；未开启 404 |
| /ai/itinerary | POST | 必须 | 生成+核验草稿（不落库）；402 无 key / 502 解析失败 |
| /weather | GET | 无 | lat/lng/start/days，返回逐日 {date, code, tmax, tmin} 或 null |

### 6.2 AI 草稿 JSON 结构

`{"days":[{"day":1,"theme":"…","items":[{"type":"SIGHT 等五取一","title":"…","note":"…","startTime":"09:00","cost":0,"searchQuery":"地点名 城市"}]}],"tips":["…"]}`——服务端解析前剥离可能的 markdown 围栏；days 非数组即判 BAD_AI_JSON。

### 6.3 外部接口

| 服务 | 端点 | 关键参数 |
|---|---|---|
| Nominatim | GET nominatim.openstreetmap.org/search | format=jsonv2, limit, accept-language=zh,en, q；UA 自报应用与联系邮箱 |
| Open-Meteo | GET api.open-meteo.com/v1/forecast | daily=weather_code,temperature_2m_max/min, timezone=auto, start_date, end_date |
| Anthropic | messages.stream | model 取环境变量（默认 claude-opus-4-8），max_tokens 8000，流式取 finalMessage |
| Carto 瓦片 | basemaps.cartocdn.com/dark_all/{z}/{x}/{y} | 前端直连，maxZoom 19 |

## 7. 数据与存储

| 模型 | 关键字段 | 说明 |
|---|---|---|
| User | email 唯一、passwordHash、displayName、isGuest | 游客与正式账号同表 |
| Trip | title、destination、startDate 可空、days、notes、shareSlug 唯一可空、ownerId | shareSlug 非空即分享中 |
| ItineraryItem | dayIndex（-1 清单）、sortOrder、type、title、note、startTime、cost、lat/lng 可空、address、verified（NONE/OK/FAIL）、source（USER/AI）、tripId | 随行程级联删除 |
| PlaceCache | query 唯一（`limit:关键词`）、resultJson、createdAt | 30 天 TTL 读取侧判断；无删除逻辑，只增不清（KI-4） |

存储介质：SQLite 单文件。开发 `backend/prisma/dev.db`，生产 `/home/ubuntu/travel/prisma/prod.db`（Prisma 相对路径以 schema 所在目录为基准）。无备份机制（规划 PR-P0-3）。

### 8.2 发布流水线（真实步骤与坑）

- 本地：`frontend && npm run build` → rsync `backend/`（排除 node_modules/dev.db/.env/dist）→ `/home/ubuntu/travel/`，rsync `frontend/dist/` → `/home/ubuntu/travel/public/` → `pm2 restart travel` → curl 冒烟。
- **坑一（真实事故）**：rsync 不含 .env 与数据库，首次部署或 schema 变更后**必须**在服务器执行 `npx prisma db push`。v0.2 上线时漏掉此步，prod.db 为 0 字节空文件，任何触库 API 直接崩进程；health 不触库故常规检查全绿，直至 2026-07-17 生成本文档核实时才发现。已补救并在 v0.2.1 加进程兜底。
- **坑二**：pm2 restart 后立即 curl 会连接失败，需等待约 3 秒。
- **坑三**：`opcstudio.cc` 域名走 Cloudflare，ssh/scp 必须用 `oracle-vm` 别名或 IP。

## 10. 质量保障与已知问题

### 10.1 质量手段

实际在用：TypeScript 双端 strict + 部署前 `tsc` 门禁；部署后人工 curl 冒烟（health/weather/guest）；浏览器端到端人工走查（本轮含注册/游客/CRUD/拖拽/搜索/分享/天气/CSV）。无自动化测试（0 个测试文件），无 CI。

### 10.2 已知问题（全部经代码核实）
